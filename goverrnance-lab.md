## Deliverable 2: Corrected Data Flow Diagram — Annotated Corrections

Since the diagram cannot be redrawn here, the corrections are described as numbered annotations that map directly to the nodes in the original diagram. These should be applied as footnote callouts on the diagram itself.

**[1] Step 1 — User Mobile App: Remove Excessive Data Collection**
Restrict the app's data collection to fields with a direct, documented purpose for credit scoring: name, phone number, Ghana Card ID, MoMo/mobile money transaction summary, and self-declared income. Remove collection of the full contact list, raw GPS coordinates, and device logs. Reason: Act 843 Section 17 requires that personal data be adequate, relevant, and not excessive relative to the purpose for which it is collected.

**[2] Between Step 2 and Step 3 — API Gateway to Raw Data DB: Insert Consent Capture and Validation Gate**
Add a consent service between the API Gateway and the Raw Data DB. Before any record is written to storage, the service checks that a valid, timestamped consent record exists for that user. Records without a consent record are rejected and the user is prompted to complete the consent flow. Reason: Act 843 requires a lawful basis for processing; explicit consent is the most appropriate basis for a fintech app collecting financial data from individuals.

**[3] Step 3 — Raw Data DB 'AllEvents': Apply Data Classification and Retention Policy**
Tag all records at write time with their classification level (Sensitive for financial and identity fields). Enforce a retention limit  for example, delete raw application data 90 days after a final loan decision, retaining only the anonymized outcome record needed for model retraining. Apply column-level encryption to Sensitive fields. Reason: Storing unclassified, unencrypted PII indefinitely with no retention schedule violates the data minimization and storage limitation requirements of Act 843.

**[4] Step 5 — Preprocessing Service: Define and Enforce Data Handling Rules**
Document which fields the Preprocessing Service reads, transforms, and passes downstream. Mask or drop fields that are not model features before the processed dataset leaves this service. Log all transformation steps with a run ID so that any data quality issue can be traced back to its source record. Reason: Without defined handling rules, sensitive fields can leak into intermediate outputs or logs that have weaker access controls than the source database.

**[5] Step 7 — Decision Service: Add Transparency Logging**
Write a structured log entry for every automated decision, recording: application ID, timestamp, model version, the top three features that drove the decision (from SHAP or equivalent explainability output), the decision outcome, and the demographic group of the applicant (where consented). Store these logs in an append-only audit table. Reason: Fully automated decisions affecting individuals' access to credit require an auditable record. Act 843 and general fairness principles require that affected individuals can, on request, receive an explanation of the basis for a decision made about them.

**[6] Step 9 — Analytics DB: Implement Data Masking and Anonymization**
Before any record is written to the Analytics DB, apply pseudonymization to direct identifiers (replace Ghana Card ID and phone number with a non-reversible hash) and suppress or generalize quasi-identifiers (replace exact age with age band, replace precise location with region). Reason: Analytics workloads typically involve broader access than operational systems. Storing unmasked PII in an analytics database increases the attack surface and makes it harder to control who can re-identify individuals.

## Deliverable 3: Summary of Review Process

The review began by mapping QuickLoan's pipeline against the standard data lifecycle stages: collection, storage, processing, use, sharing, and deletion. At each stage, the question was whether the data flowing through matched what was actually needed for the stated purpose  credit scoring  and whether the controls in place were proportionate to the sensitivity of the data.

Classification was the first analytical tool applied. The data QuickLoan collects  Ghana Card numbers, mobile money transaction histories, phone numbers, and (in the current flawed pipeline) contact lists and GPS logs  sits at the Sensitive tier. That classification immediately sets the baseline for what controls are non-negotiable: encryption at rest and in transit, strict access controls, consent before collection, and a defined retention limit. Once the data was classified, the gaps became visible by comparison: no encryption, no consent gate, no retention schedule, and no access segmentation between the operational database and the analytics database.

The data lifecycle framing then identified where in the pipeline each gap sat. Excessive collection is a collection-stage failure. Missing consent is a collection-to-storage transition failure. No masking in the Analytics DB is a use-stage failure. No logging at the Decision Service is a use-stage failure with direct fairness consequences. This stage-by-stage approach prevented the review from treating the pipeline as a single undifferentiated problem and made it possible to assign a specific, bounded fix to each node.

The proposed metric  Demographic Approval Rate Disparity addresses the fairness gap directly. Because it is calculated from the decision logs that the corrected pipeline will now produce, it creates a feedback loop: the logging correction at Step 7 generates the data, and the DARD metric converts that data into a number that compliance staff can act on each month. A rising DARD is an early signal that the model's behavior is diverging across groups, which triggers investigation before the disparity becomes large enough to constitute a systematic violation. Publishing DARD in an internal governance report also gives QuickLoan a concrete record of due diligence, which is relevant if the Data Protection Commission ever requests evidence of fair processing.
